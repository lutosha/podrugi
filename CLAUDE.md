# Podrugi

Социальная сеть для девушек Лондона (домен `podrugi.co.uk`). Основательница — нетехнический человек, весь проект собирается через Claude Code по учебному курсу (вайбкодинг). Объясняй шаги просто, без лишнего жаргона.

## Стек

- **Frontend**: чистые HTML/CSS/JS, без сборщика и фреймворка. `frontend/index.html`, `frontend/style.css`, `frontend/script.js`.
- **Backend**: Node.js + Express, `backend/index.js`. Один файл, без роутеров/контроллеров — не разбивать на файлы, пока проект маленький.
- **База данных**: PostgreSQL через Prisma (`backend/prisma/schema.prisma`). Модель `User` пока единственная.
- **Auth**: bcrypt (хеш пароля) + JWT (7 дней). Middleware `requireAuth` в `backend/index.js`.

## Хостинг и деплой

- Репозиторий: github.com/lutosha/podrugi (владелец аккаунта — пользователь, лично).
- **Frontend** — Netlify, publish directory `frontend`, автодеплой при пуше в `main`. Живой адрес: https://podrugi.netlify.app
- **Backend** — Railway, Root Directory `backend`, автодеплой при пуше в `main`. Живой адрес: https://podrugi-production.up.railway.app
- **База** — Postgres в том же Railway-проекте, что и backend.
- Прямой `git push` из терминала не работает — на компьютере не настроена авторизация GitHub. Пуши делаются через подключённый GitHub MCP-инструмент (`push_files` и т.д.), не через `git push`.
- Домен `podrugi.co.uk` ещё не подключён к Netlify/Railway (это Урок 11 — в процессе).

## Переменные окружения

- Локально: `backend/.env` (в Git не попадает) — `DATABASE_URL` (публичный адрес Railway-базы, не `.railway.internal`, иначе не подключится с компьютера), `JWT_SECRET`.
- На Railway: те же переменные заданы напрямую в Variables сервиса (не через файл). `DATABASE_URL` там — ссылка на переменную сервиса Postgres, `JWT_SECRET` — сгенерирован Railway отдельно, **отличается** от локального, и это нормально.

## Известные грабли (уже наступили и исправлены — не повторять)

- **Prisma 7** требует явный driver adapter (`@prisma/adapter-pg`) — `new PrismaClient()` без него падает.
- `prisma` и `dotenv` должны быть в `dependencies`, а не `devDependencies` — иначе на Railway (production install) не выполнится `prisma generate` и деплой упадёт с `Cannot find module '.prisma/client/default'`. Есть скрипт `postinstall: prisma generate` для подстраховки.
- Скрипт `start` в `backend/package.json` НЕ должен использовать `--env-file=.env` (падает на Railway, там нет .env-файла, только переменные окружения). Флаг `--env-file` оставлен только в `dev`.
- CORS в `backend/index.js` — список `ALLOWED_ORIGINS` нужно обновлять руками при добавлении нового домена (сейчас: `podrugi.co.uk`, `www.podrugi.co.uk`, `podrugi.netlify.app`, плюс localhost любого порта — для разработки).
- `frontend/script.js`: `API_BASE_URL` сам переключается между локальным бэкендом и Railway в зависимости от `location.hostname` — не хардкодить один адрес.

## Дорожная карта фич (после уроков)

План лежит в `.claude/plans` (не в репозитории) — кратко: A (укрепить auth) → B (посты/лента) → C (жалобы/блокировка/роли, обязательно до реальных юзеров) → D (подписки) → E (сообщения) → F (районы/фильтры) → G (рост, отложено).

Пройдено:
- **Фаза A** — rate limiting (`express-rate-limit`, 10 запросов/15 мин на `/api/register` и `/api/login`), валидация входных данных через `zod`, чекбокс «Мне есть 18» при регистрации (поле `ageConfirmed` в модели `User`, обязательно `true` для регистрации).
- **Фаза B** — модель `Post` (content, area, authorId). `POST /api/posts` (только для вошедших), `GET /api/posts` (гость видит последние 3, вошедший — 50), `GET /api/users/:id` (публичный профиль). Фронтенд: реальная лента вместо демо-карточек, форма публикации поста видна только вошедшим. В базе на Railway сейчас есть тестовые пользователи/посты, оставленные намеренно (пользователь попросила не чистить) — не удивляться, если увидишь их в ленте.
- **Фаза B (расширение) — три типа контента.** `Post.type`: `POST` / `ANNOUNCEMENT` / `EVENT` (+ `eventDate` для событий). Модели `Comment` (комментарии — на всех трёх типах) и `Rsvp` (`GOING`/`MAYBE`, только для `EVENT`, unique по `postId+userId`). Терминология в интерфейсе (важно не путать при дальнейшей работе):
  - **Пост** — обычный вопрос/шеринг («Посоветуйте косметолога в Ealing»).
  - **Объявление** — запрос/просьба («Ищу партнёра для тенниса в Fulham»).
  - **Событие** — с датой и RSVP («Рисуем картины в парке 1 августа»).
  - Связи между людьми называются **«Подруги»** (не «подписки»/«друзья») — сознательный выбор под бренд.
  - Эндпоинты: `POST /api/posts/:id/comments`, `POST /api/posts/:id/rsvp`. Фронтенд: вкладки выбора типа над формой, бейдж типа на карточке, кнопки RSVP с счётчиками, комментарии под каждым постом.

- **Фаза C** — безопасность сообщества. `User.role` (`USER`/`MODERATOR`/`ADMIN`, аккаунт `lutosha@gmail.com` — `ADMIN`). Модели `Report` (жалоба на пост или пользователя, статус `PENDING`/`RESOLVED`/`DISMISSED`) и `Block` (кто кого заблокировал, unique пара). `GET /api/posts` теперь исключает посты пользователей из чёрного списка (в обе стороны — если я заблокировала или меня заблокировали). Эндпоинты: `POST /api/reports`, `GET /api/reports` + `PATCH /api/reports/:id` (только `MODERATOR`/`ADMIN`, middleware `requireModerator`), `POST/DELETE /api/block/:userId`. Фронтенд: кнопки «Пожаловаться»/«Заблокировать» под каждым постом (не у своих постов), ссылка «Модерация» в шапке видна только модераторам/админам, отдельная страница `frontend/moderation.html` + `moderation.js` (самостоятельный скрипт, НЕ подключает основной `script.js` — там нет нужных DOM-элементов). Плюс статичные `frontend/guidelines.html` и `frontend/privacy.html`, ссылки на них в футере.

- **Мобильная адаптация (приоритет продукта — пользователи в основном на телефонах).** Проверено и исправлено на 320px и 375px: шапка теперь переносится (`flex-wrap`) вместо обрезания на узких экранах; все кнопки/поля ввода — `min-height: 44px` (кнопки) / `40px` (мелкие элементы вроде RSVP, комментариев, жалоб) под палец; текстовые поля в комментариях/жалобах увеличены до `font-size: 16px` (меньший размер вызывает нежелательный зум при фокусе в Safari на iOS); модалка входа/регистрации — `max-height: 90vh` с прокруткой. Медиа-запросы на `min-width: 480px` возвращают увеличенные отступы/шрифты для десктопа. При добавлении новых элементов интерфейса — сразу проверять на 320-375px, не откладывать на потом.

- **Фаза E** — личные сообщения. Модель `Message` (senderId, recipientId, content). `POST /api/messages/:userId` (нельзя себе, нельзя если есть блокировка в любую сторону), `GET /api/messages/:userId` (переписка), `GET /api/conversations` (список бесед с последним сообщением). Фронтенд: ссылка «Написать» на посте (не у своих постов) → `frontend/messages.html` + `messages.js` (самостоятельный скрипт, как `moderation.js`) — список бесед без параметров, конкретная переписка через `?to=ID&name=Имя`. Ссылка «Сообщения» в шапке для всех вошедших. Сообщения открыты для всех зарегистрированных (не только «подруг») — защита от спама через блокировку/жалобы (Фаза C).

- **Фаза D (подруги) — реализована.** Изначально была пропущена (сообщения открыты всем без ограничения), но решение пересмотрено — «Подруги» нужны как самостоятельная фича, не только как гейт для сообщений. Модель `Follow` (followerId, followingId, unique пара). Эндпоинты: `POST/DELETE /api/follow/:userId` (нельзя подписаться на себя), `GET /api/friends` (список тех, на кого подписан текущий пользователь). `GET /api/posts` теперь возвращает `authorIsFollowed` (булево) для каждого поста. Фронтенд: кнопка «Добавить в подруги» / «В подругах ✓» на каждом посте (не у своих), ссылка «Подруги» в шапке для всех вошедших → `frontend/friends.html` + `friends.js` (самостоятельный скрипт, как `moderation.js`/`messages.js`) — список подруг с кнопкой «Написать» на каждую (ведёт в `messages.html?to=ID&name=Имя`). Протестировано в браузере на desktop и mobile (375px): подписка/отписка через кнопку на посте, список на странице «Подруги», переход в переписку — всё работает.

- **Фаза F (районы/фильтры) — реализована**, фильтр по району сначала был `<select>`, потом переделан в чипы (см. ниже «UX-фаза 1»). Эндпоинт `GET /api/areas` возвращает список уникальных непустых значений `area` из постов (отсортирован). `GET /api/posts` принимает query-параметр `?area=` и фильтрует по точному совпадению.

- **Профиль пользователя.** Страница `frontend/profile.html` + `profile.js` (самостоятельный скрипт, как остальные вспомогательные страницы) — принимает `?id=X`. Если это свой профиль — показывает форму редактирования имени/района/фото (`PATCH /api/profile`); если чужой — показывает кнопки «Подруги/Написать/Пожаловаться/Заблокировать» (переиспользует существующие эндпоинты follow/block/reports) и список постов этого автора через query-параметр `GET /api/posts?authorId=X` (учитывает блокировки: если автор в чёрном списке — возвращает пустой массив вместо утечки постов). Статус «в подругах» на чужом профиле определяется через `GET /api/friends`. Имя автора на карточке поста в `script.js` и имя в списке подруг в `friends.js` — кликабельные ссылки на `profile.html?id=X`.

- **Мобильная навигация по мотивам Facebook/Meetup/Instagram (после анализа 10 похожих соцсетей).** Заменили верхнее меню «Профиль ▾» на постоянную нижнюю таб-панель (`.bottom-nav` в `style.css`, одинаковая разметка на `index.html`/`friends.html`/`messages.html`/`profile.html`/`moderation.html`): Лента / Подруги / Написать (+) / Сообщения / Профиль. Видна только вошедшим (`body.has-bottom-nav` добавляет отступ снизу под фиксированную панель). Иконка вкладки «Профиль» показывает настоящий аватар пользователя (или букву-инициал), а не статичную иконку — обновляется при логине и при сохранении нового фото. Каждая страница (кроме `index.html`, где логика уже есть в основном `fetchProfile()`) содержит свою копию `initBottomNav()` — по той же причине, что и дублирование `API_BASE_URL`: скрипты самостоятельные и не подключают друг друга.
  «Выйти» и «Модерация» (для модераторов/админов) переехали с верхнего меню на страницу своего профиля (`profile.js` → `buildOwnProfile`), а не висят в панели — так же, как в Facebook/Instagram, где настройки/выход находятся на экране профиля, а не в основной навигации.
  Форма публикации поста (`#postForm`) стала модалкой (`#postModal`, тот же паттерн, что и `#authModal`), открывается кнопкой «Написать» в нижней панели — на `index.html` кнопка перехватывает клик и открывает модалку сразу, на остальных страницах это обычная ссылка на `index.html?compose=1`, которая после логики `fetchProfile()` открывает модалку и подчищает URL через `history.replaceState`. Лента больше не «прыгает» из-за всегда открытой формы над постами.
  **Известный баг, который встретился и был исправлен во всех четырёх копиях `initBottomNav()`**: функция снимала `hidden` только в неудачном сценарии (добавляла обратно), но никогда не убирала класс `hidden` при успешной загрузке профиля — из-за этого нижняя панель оставалась невидимой даже у вошедших пользователей. Чинится строкой `bottomNav.classList.remove('hidden')` сразу после успешного `fetch('/api/profile')`. Если будешь копировать этот паттерн для новой страницы — не забыть эту строку.

- **Реальные фото профиля вместо инициалов.** Поле `User.avatar` (String?, nullable) хранит **base64 data URI**, а не ссылку на файл — сознательное решение, чтобы не заводить отдельный сервис хранения файлов (Railway не хранит файлы между деплоями, S3/Cloudinary — это новый платный сервис и лишняя инфраструктура для MVP). Фото сжимается на фронтенде перед отправкой: `profile.js` через `<canvas>` обрезает и уменьшает до 160×160, кодирует в JPEG с качеством 0.8 (`resizeImageToDataUrl`) — итоговый размер обычно 10–30 КБ, укладывается в лимит `updateProfileSchema.avatar` (300 000 символов) и не раздувает базу. Если фото нет — везде показывается кружок с первой буквой имени (как раньше), это общий паттерн `buildAvatarElement`/`setAvatarContent`, продублированный в `script.js`, `profile.js`, `friends.js`, `messages.js` (те же причины дублирования, что и с `API_BASE_URL`). Аватар подключён к: карточкам постов, списку подруг, списку переписок, шапке профиля, иконке вкладки «Профиль» в нижней навигации.

- **UX-фаза 1 (после анализа 10, затем ещё 20 похожих сервисов) — реализована.** Порядок был согласован с пользователем как «фаза 1: дешёвое и без изменений в БД → фаза 2: вовлечение → фаза 3: события → фаза 4: рост связей»; сделана фаза 1 целиком:
  - **Значок-флаг вместо кнопок «Пожаловаться»/«Заблокировать».** В `buildActionsSection` (`frontend/script.js`) и `buildOtherProfile` (`frontend/profile.js`) обе кнопки заменены на один `.flag-btn` (🚩), открывающий маленькое выпадающее меню `.flag-menu` с теми же двумя действиями (сама логика report/block не менялась). Сравнивали с long-press как в Telegram — отказались: на вебе жест ненадёжен (конфликт со скроллом, нет аналога на desktop). Follow/«Написать» остались обычными видимыми кнопками — переполнение было именно от модераторских действий, не от них.
  - **Чипы вместо `<select>` для фильтра районов** (паттерн Nextdoor) — `#areaFilter` теперь горизонтально скроллящийся ряд `.area-chip`, тот же принцип, что и `.type-tabs`. Состояние — переменная `selectedArea` вместо `.value` у `<select>`.
  - **Переключатель «Все»/«Подруги» в ленте** — `#feedScope` (`.scope-btn`), виден только вошедшим. На бэкенде `GET /api/posts` получил query-параметр `?following=1`, который фильтрует `where.authorId` по списку тех, на кого подписан текущий пользователь (за вычетом чёрного списка) — сделано на сервере, а не на клиенте, чтобы не терять посты за пределами `take: 50`.
  - **Поле «о себе» (bio) в профиле.** `User.bio` (String?, максимум 300 символов на бэкенде). Textarea в форме редактирования профиля, отображается под районом на `renderProfileHeader` — общая функция для своего и чужого профиля, так что бэкенд `GET /api/users/:id` тоже отдаёт `bio` всем.
  - Осознанно НЕ стали делать верификацию личности через селфи (паттерн Bumble/Airbnb) — пользователь решила, что это подняло бы порог входа для тех, кто пока не уверен в сервисе; ограничились обычной загрузкой фото без проверки.
