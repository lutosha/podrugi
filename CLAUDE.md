# Podrugi

Социальная сеть для девушек Лондона (домен `podrugi.co.uk`). Основательница — нетехнический человек, весь проект собирается через Claude Code по учебному курсу (вайбкодинг). Объясняй шаги просто, без лишнего жаргона.

## Стек

- **Frontend**: чистые HTML/CSS/JS, без сборщика и фреймворка. `frontend/index.html`, `frontend/style.css`, `frontend/script.js`.
- **Backend**: Node.js + Express, `backend/index.js`. Один файл, без роутеров/контроллеров — не разбивать на файлы, пока проект маленький.
- **База данных**: PostgreSQL через Prisma (`backend/prisma/schema.prisma`). Модель `User` пока единственная.
- **Auth**: bcrypt (хеш пароля) + JWT (7 дней). Middleware `requireAuth` в `backend/index.js`.

## Хостинг и деплой

- Репозиторий: github.com/lutosha/podrugi (владелец аккаунта — пользователь, лично).
- **Frontend** — Netlify, publish directory `frontend`, автодеплой при пуше в `main`. Живой адрес: https://podrugi.co.uk (запасной технический адрес Netlify: https://podrugi.netlify.app).
- **Backend** — Railway, Root Directory `backend`, автодеплой при пуше в `main`. Живой адрес: https://api.podrugi.co.uk (запасной технический адрес Railway: https://podrugi-production.up.railway.app — оставлен в `ALLOWED_ORIGINS`/CORS, но фронтенд на него больше не ссылается).
- **База** — Postgres в том же Railway-проекте, что и backend.
- Домен `podrugi.co.uk` подключён к обоим сервисам, SSL выпущен и работает на всех адресах (проверено `curl` — и `podrugi.co.uk`, и `api.podrugi.co.uk` отдают валидный HTTPS). Раньше здесь была запись «домен ещё не подключён» — устарела, оставлена в истории коммитов как урок: **эту секцию надо обновлять по факту, а не один раз в начале проекта**.
- Прямой `git push` из терминала не работает — на компьютере не настроена авторизация GitHub (проверено: `git push` спрашивает пароль, а не токен через keychain). Пуш делается через **GitHub Desktop** (кнопка «Push origin», либо «Force push origin» через дропдаун ▾ у кнопки, если ветки разошлись).

## Переменные окружения

- Локально: `backend/.env` (в Git не попадает) — `DATABASE_URL` (публичный адрес Railway-базы, не `.railway.internal`, иначе не подключится с компьютера), `JWT_SECRET`.
- На Railway: те же переменные заданы напрямую в Variables сервиса (не через файл). `DATABASE_URL` там — ссылка на переменную сервиса Postgres, `JWT_SECRET` — сгенерирован Railway отдельно, **отличается** от локального, и это нормально.

## Известные грабли (уже наступили и исправлены — не повторять)

- **Prisma 7** требует явный driver adapter (`@prisma/adapter-pg`) — `new PrismaClient()` без него падает.
- `prisma` и `dotenv` должны быть в `dependencies`, а не `devDependencies` — иначе на Railway (production install) не выполнится `prisma generate` и деплой упадёт с `Cannot find module '.prisma/client/default'`. Есть скрипт `postinstall: prisma generate` для подстраховки.
- Скрипт `start` в `backend/package.json` НЕ должен использовать `--env-file=.env` (падает на Railway, там нет .env-файла, только переменные окружения). Флаг `--env-file` оставлен только в `dev`.
- CORS в `backend/index.js` — список `ALLOWED_ORIGINS` нужно обновлять руками при добавлении нового домена (сейчас: `podrugi.co.uk`, `www.podrugi.co.uk`, `podrugi.netlify.app`, плюс localhost любого порта — для разработки).
- Каждый самостоятельный JS-файл (`script.js`, `profile.js`, `friends.js`, `messages.js`, `moderation.js`) хранит свою копию `API_BASE_URL`, переключается между локальным бэкендом и `https://api.podrugi.co.uk` в зависимости от `location.hostname` — не хардкодить один адрес и не забывать поправить все 5 копий, если адрес бэкенда снова сменится.

### Редизайн ленты по мокапу (3 секции)

Повод: пользователь захотела поработать над дизайном «экран за экраном», сама сделала HTML/CSS/JS-мокап в параллельном чате (получил краткое описание нашей текущей структуры ленты) и попросила перенести дизайн к нам. Уточнили с ней объём тремя вопросами — она выбрала: точная палитра мокапа (не старые 4 hex-кода), «Ищу компанию» = наш существующий тип поста `ANNOUNCEMENT` в новом визуальном представлении (не новая сущность в БД), и полный объём (все 3 секции + теги + фото-обложки событий, не только смена цветов).

- **Структура ленты.** `index.html` вместо единого списка постов с чипом-фильтром типа теперь имеет 3 секции подряд: **«Ищу компанию»** (только `ANNOUNCEMENT`, сетка коротких карточек-ссылок на профиль автора — аватар, статус-точка, первая строка текста, район), **«Мероприятия»** (только `EVENT`, карточки с градиентной «обложкой» вместо реальных фото — см. ниже), **«Посты»** (только `POST`, прежний вид карточки). Рендер — `renderFeed(posts)` в `script.js`, диспетчер к трём функциям `renderCompanyGrid`/`renderEventCards`/`renderPosts`; серверный фильтр по типу (`?type=`) убран, весь сплит теперь на фронтенде из одного ответа `GET /api/posts`.
- **Фильтр района — выпадающий список вместо чипов.** `.district-select`/`.district-trigger`/`.district-menu` (открывается по клику, закрывается по клику вне себя или повторному клику, отмечает выбранный пункт галочкой) заменил горизонтальный ряд `.area-chip` над лентой — чипы остаются в форме поста (выбор района при публикации), только фильтр ленты сверху сменил вид.
- **«Загрузить ещё».** И «Ищу компанию», и «Мероприятия» по умолчанию показывают по 3 карточки (`companyVisible`/`eventsVisible`, изначально `3`), кнопка `.section-load-more` ставит `Infinity` и перерисовывает из закэшированного `latestPosts` без повторного запроса к серверу. «Посты» без этого ограничения — там весь список сразу, как раньше.
- **Теги событий.** `Post.tags` (`String[] @default([])`, миграция `20260723194202_add_post_tags`) — до 5 тегов, до 20 символов каждый (`postSchema` в `backend/index.js`). Поле есть только у `EVENT`: `#postTags` в форме поста (текст через запятую, парсится в массив на фронтенде), показывается/скрывается вместе с датой и лимитом участников через общий `setPostType(type)`. На карточке события — цветные плашки `.discover-tag` поверх обложки.
- **Градиентные «обложки» вместо реальных фото.** У нас нет загрузки фото для постов (только для аватара профиля, см. ниже про `avatar`) — заводить это ради одной карточки события было бы лишней инфраструктурой. Вместо реального фото `.discover-photo` красится одним из трёх заготовленных CSS-градиентов (`COVER_GRAD` в `script.js`, детерминированно по `post.id % 3` — одно и то же событие всегда получает один и тот же градиент, а не рандомный при каждой перерисовке).
- **RSVP-аватары «идут».** `postInclude.rsvps` расширен — теперь отдаёт не только `userId`/`status`, но и `user: { name, avatar }`, чтобы показать мини-стек аватаров тех, кто идёт (раньше на карточке было только число).
- **Сознательно упрощено/не перенесено из мокапа:** кнопки «поделиться» нет (некуда — публичных пермалинков на отдельный пост не существует); комментарии показываются сразу под постом целиком, а не по клику «показать комментарии» (в мокапе — раскрывающийся блок, у нас пока на реальных данных комментариев мало, разворачивать нечего).

### Доработка редизайна: «оболочка» сайта (после фидбека «не похоже на референс»)

Первый проход редизайна (выше) перенёс только контент секций ленты, но не общую рамку страницы — пользователь сразу заметила разницу и явно попросила «дизайн ровно как в референсе». Сравнили её мокап и наш сайт бок о бок (открыла оба в браузере) и нашли реальный пробел: не был перенесён контейнер-«телефон», шапка, hero-декорация, нижняя панель с плавающей кнопкой и тёмная тема. Уточнили один момент, где мокап расходится с уже принятым решением — лого: **пользователь подтвердила оставить свой `logo.svg`** (не менять на текстовый вордмарк из мокапа).

- **Контейнер `.app`** (`frontend/style.css`) — новый `<div class="app" id="app">` оборачивает `header`+`main`+`footer`+`bottomNav` на каждой из 6 страниц (модалки и `<script>` — вне него, как в мокапе). `max-width: 480px`, по центру; на экранах шире 560px добавляются отступ, скругление 16px и тень — сайт выглядит как «карточка-телефон» на десктопе, а не растянутая на всю ширину страница. Сам `<body>`/`<html>` не трогали (остаются обычным скролл-контейнером) — контейнер только визуальный, не влияет на скролл/sticky.
- **Тёмная тема.** Добавлен блок `@media (prefers-color-scheme: dark)` с тёмными значениями всех CSS-переменных (из мокапа) — переключается автоматически по системной теме, без отдельного тумблера в интерфейсе. Лого `logo.svg` в тёмной теме получает `filter: invert(1) hue-rotate(180deg)` — сам файл нарисован чёрным текстом на прозрачном фоне, без фильтра был бы невиден на тёмном фоне; фильтр — единственный способ адаптировать растровые цвета внешнего `<img>` под тему, не трогая сам SVG-файл (он «прислан пользователем», редактировать не стали).
- **Шапка.** `position: sticky` + полупрозрачный фон с `backdrop-filter: blur`, компактные отступы. Колокольчик уведомлений (`.notif-bell`) и кнопки гостя («Войти»/«Регистрация») перерисованы под стиль мокапа: колокольчик — кружок с обводкой и inline-SVG (эмодзи 🔔 убран), «Войти» — прозрачная кнопка с шалфейной обводкой, «Регистрация»/все `.btn-primary` — розовая заливка (`var(--accent)`) вместо чёрной (`var(--ink)`, как было раньше — это единственная реальная нестыковка мокапа: акцентный цвет кнопок в мокапе розовый, а не чёрный, хотя чёрный остался в палитре текста/заголовков логотипа).
- **Hero.** Декоративный SVG-силуэт города (`.hero-skyline`, полупрозрачный, тот же контур, что в мокапе) добавлен фоном; текст стал левым (было — по центру), сузился (`max-width: 38ch` у параграфа). Блок целиком скрывается для вошедших (`#heroSection`, toggle в `showLoggedIn`/`showLoggedOut` в `script.js`) — в мокапе hero виден только гостю, у нас раньше был виден всегда.
- **Нижняя навигация → sticky tabbar + плавающая кнопка (FAB).** `.bottom-nav` был `position: fixed` (перекрывал весь viewport, что на десктопе выглядело бы криво поверх узкой карточки `.app`) — стал `position: sticky; margin-top: auto`, чтобы жить внутри flex-колонки `.app` и не вылезать за её тень/скругление. Средняя кнопка «Написать» (`.bottom-nav-compose`) была обычным пунктом меню с текстом — теперь отдельный кружок 52px, приподнятый над панелью (`margin-top: -26px`), без подписи, только SVG-плюс — классический паттерн FAB. Эмодзи-иконки остальных 4 пунктов (🏠👭💬👤) заменены на inline-SVG line-иконки (аватар профиля — единственное исключение, там как и раньше настоящее фото/буква инициала, а не иконка, это осознанный выбор персонализации). Правки внесены на всех 6 страниц (`index.html`/`friends.html`/`messages.html`/`profile.html`/`moderation.html`/`notifications.html`) — разметка шапки/нижней панели там дублируется по той же причине, что и остальные общие паттерны (`API_BASE_URL` и т.д.). Старый CSS-хак `body.has-bottom-nav { padding-bottom: 76px }` (компенсация для фиксированной панели) убран — sticky-панели он больше не нужен; JS-вызовы `classList.add/remove('has-bottom-nav')` в 5 вспомогательных JS-файлах оставлены (безвредны без соответствующего CSS-правила, не стали чистить ради экономии времени).
- **RSVP-кнопки** получили SVG-иконки (галочка у «Иду», знак вопроса у «Возможно») перед текстом счётчика — в первом проходе редизайна были только текстовые.
- **Модалки.** Вход/Регистрация — раньше одна форма с текстовым заголовком, который менялся по JS (`modalTitle.textContent`); теперь заголовок статичный («Подруги», курсивным шрифтом) + переключатель `.auth-tabs` (Вход/Регистрация) с подчёркиванием активной вкладки — тот же принцип, что и `.type-tabs` в форме поста. Вкладка формы поста «Объявление» переименована в **«Ищу компанию»** (тот же тип `ANNOUNCEMENT` на бэкенде, просто текст в интерфейсе теперь совпадает с названием секции ленты, а не техническим термином).
- Отдельная страница профиля (`profile.js`) и вспомогательные страницы (`friends.js`/`messages.js`/`moderation.js`/`notifications.js`) содержательно не менялись — их правки ограничены тем же контейнером `.app` и шапкой/навигацией, что и везде.

## Дорожная карта фич (после уроков)

План лежит в `.claude/plans` (не в репозитории) — кратко: A (укрепить auth) → B (посты/лента) → C (жалобы/блокировка/роли, обязательно до реальных юзеров) → D (подписки) → E (сообщения) → F (районы/фильтры) → G (рост, отложено).

Пройдено:
- **Фаза A** — rate limiting (`express-rate-limit`, 10 запросов/15 мин на `/api/register` и `/api/login`), валидация входных данных через `zod`, чекбокс «Мне есть 18» при регистрации (поле `ageConfirmed` в модели `User`, обязательно `true` для регистрации).
- **Фаза B** — модель `Post` (content, area, authorId). `POST /api/posts` (только для вошедших), `GET /api/posts` (гость видит последние 3, вошедший — 50), `GET /api/users/:id` (публичный профиль). Фронтенд: реальная лента вместо демо-карточек, форма публикации поста видна только вошедшим. В базе на Railway сейчас есть тестовые пользователи/посты, оставленные намеренно (пользователь попросила не чистить) — не удивляться, если увидишь их в ленте.
- **Фаза B (расширение) — три типа контента.** `Post.type`: `POST` / `ANNOUNCEMENT` / `EVENT` (+ `eventDate` для событий). Модели `Comment` (комментарии — на всех трёх типах) и `Rsvp` (`GOING`/`MAYBE`, только для `EVENT`, unique по `postId+userId`). Терминология в интерфейсе (важно не путать при дальнейшей работе):
  - **Пост** — обычный вопрос/шеринг («Посоветуйте косметолога в Ealing»).
  - **Объявление** — запрос/просьба («Ищу партнёра для тенниса в Fulham»).
  - **Событие** — с датой и RSVP («Рисуем картины в парке 1 августа»).
  - Связи между людьми называются **«Подруги»** (не «подписки»/«друзья») — сознательный выбор под бренд.
  - Эндпоинты: `POST /api/posts/:id/comments`, `POST /api/posts/:id/rsvp`. Фронтенд: вкладки выбора типа над формой, бейдж типа на карточке, кнопки RSVP с счётчиками, комментарии под каждым постом.

- **Фаза C** — безопасность сообщества. `User.role` (`USER`/`MODERATOR`/`ADMIN`, аккаунт `lutosha@gmail.com` — `ADMIN`). Модели `Report` (жалоба на пост или пользователя, статус `PENDING`/`RESOLVED`/`DISMISSED`) и `Block` (кто кого заблокировал, unique пара). `GET /api/posts` теперь исключает посты пользователей из чёрного списка (в обе стороны — если я заблокировала или меня заблокировали). Эндпоинты: `POST /api/reports`, `GET /api/reports` + `PATCH /api/reports/:id` (только `MODERATOR`/`ADMIN`, middleware `requireModerator`), `POST/DELETE /api/block/:userId`. Ссылка — обычный `<a href=...>` без `download`, добавлена только в `script.js` (лента), не на `profile.js` (та же логика, что и с реакциями — профиль сознательно без интерактива).
  - **Лимит участников.** `Post.maxParticipants` (Int?, только для `EVENT`). Задаётся необязательным полем в форме поста (видно только на вкладке «Событие», как и дата). `POST /api/posts/:id/rsvp` считает текущих `GOING` (кроме самого себя — иначе повторное подтверждение своего же RSVP ошибочно блокировалось бы) и отклоняет с 400, если лимит достигнут. Фронтенд показывает «Иду (3/10)» вместо «Иду (3)» и блокирует кнопку «Иду», если мест нет и сама пользователь(ница) ещё не записана (у «Возможно» лимита нет).

- **Фаза 4 («люди рядом») — реализована.** `GET /api/nearby` (требует авторизации) отдаёт до 20 пользователей с тем же `borough`, что и у текущего пользователя, минус сам пользователь, минус те, на кого уже подписан, минус заблокированные (в обе стороны, тот же `getBlockedUserIds`/`getFollowingIds`, что уже использовались в `/api/posts`), сортировка — по `createdAt desc` (сначала новые аккаунты). Фронтенд: новая секция «Люди рядом» на `frontend/friends.html` под списком подруг (`friends.js`, те же карточки `.post-card`, что и в списке подруг, только с кнопкой «Добавить в подруги» вместо «Написать»). После успешной подписки карточка убирается из «Людей рядом» без перезагрузки страницы и список «Подруги» обновляется тем же вызовом `loadFriends()`. Пустое состояние — «Пока никого не нашлось в твоём районе» (ожидаемо часто на старте, пока в каждом borough мало пользователей). Neighbourhood/соседние borough в подбор не входят — сознательно отложено вместе с остальным neighbourhood-функционалом (см. «Локация: borough вместо свободного текста»).

- **Фаза G (частично) — уведомления и фильтр по типу поста реализованы.**
  - **Уведомления.** Уточнили с пользователем объём: список внутри приложения, не push-уведомления браузера (push — отдельно, вместе с PWA, если до этого дойдёт). Модель `Notification` (`type`: `FOLLOW`/`COMMENT`/`REACTION`/`RSVP`, `recipientId`, `actorId`, `postId?`, `read`). Создаётся напрямую в существующих эндпоинтах — `POST /api/follow/:userId`, `/api/posts/:id/comments`, `/api/posts/:id/react`, `/api/posts/:id/rsvp` — с двумя правилами: не уведомлять о действиях над собственным контентом/собой (проверка `actorId !== recipientId`) и не дублировать уведомление на idempotent-переотправку того же самого действия (для FOLLOW/REACTION — только если записи ещё не было; для RSVP — только если статус реально изменился, иначе повторный клик по уже выбранному «Иду» заспамил бы уведомлениями). `GET /api/notifications` — последние 30 с данными автора действия и превью поста. `POST /api/notifications/seen` — проставляет `read: true` всем непрочитанным. `GET /api/unread` расширен третьим полем `notifications`.
  - Точка входа — не бейдж в нижней навигации (там уже 5 пунктов, добавлять шестой было бы тесно для мобильного), а **колокольчик в шапке** (`#notificationsLink`, эмодзи 🔔), виден только вошедшим, с той же точкой `.nav-badge`, что и у бейджей на Подругах/Сообщениях. Ведёт на новую `frontend/notifications.html` + `notifications.js` (тот же самостоятельный паттерн, что и `friends.js`/`messages.js`/`moderation.js`). Разметка колокольчика продублирована в шапке всех 6 HTML-страниц (включая саму `notifications.html` — для единообразия, как и с нижней навигацией, где активная вкладка всё равно показывает саму себя).
  - **Фильтр по типу поста.** Чипы «Все типы / Посты / Объявления / События» на `index.html`, тот же класс `.area-chip`, что и у чипов района (не плодили новый CSS). Бэкенд: `GET /api/posts` принимает `?type=POST|ANNOUNCEMENT|EVENT`, невалидное значение просто игнорируется. Работает вместе с уже существующей логикой «поднять предстоящие события, скрыть прошедшие» без конфликтов — если тип не EVENT, эта логика просто не находит событий и молча ничего не переупорядочивает.
  - **Грабли: локальный сервер не подхватывает правки на лету.** `node index.js` (без nodemon) не перечитывает файл — после правок бэкенда обязательно `pkill` + перезапуск перед тестированием, иначе гоняешь тесты против старого кода и ловишь «баги», которых на самом деле уже нет в файле.

- **Фаза G (продолжение) — PWA реализован.**
  - `frontend/manifest.json` — name/icons/`display: standalone`/цвета из палитры Warm blush (`background_color: #faf6f1`, `theme_color: #c6597a`). Иконки (`assets/icon-192.png`, `assets/icon-512.png`) — **не нарисованы с нуля**, а вырезаны из уже существующего `logo.svg` (сердечко над «I», второй `<path>` в файле, залитое `#FC809F`), отцентрировано на фоне `#faf6f1` — тот же принцип, что и с самим логотипом («логотип не мой, прислан пользователем» — не придумывать новый визуальный знак самой, если можно переиспользовать уже согласованный элемент бренда). Если понадобится другая иконка — просто заменить эти два PNG, манифест их не генерирует.
  - `<link rel="manifest">` + `<meta name="theme-color">` + `<link rel="apple-touch-icon">` добавлены во все 6 HTML-страниц.
  - `frontend/sw.js` — service worker, кэширует все статические файлы (HTML/CSS/JS/manifest/иконки) при установке, стратегия fetch — stale-while-revalidate (отдаём из кэша сразу, параллельно обновляем в фоне; если сети нет — отдаём кэш). Не трогает запросы не на свой origin (т.е. все вызовы к `api.podrugi.co.uk` идут напрямую в сеть, не кэшируются — это динамические данные, кэшировать их через SW было бы неправильно). Регистрируется маленьким инлайн-`<script>` перед подключением основного JS-файла на каждой странице (не через сам JS-файл — сознательно, чтобы не плодить шестую копию одной и той же логики регистрации в каждом самостоятельном скрипте).
  - Проверено в браузере: manifest отдаётся и парсится, `navigator.serviceWorker.getRegistrations()` показывает активную регистрацию, все 20 файлов из precache-списка реально лежат в `caches`, иконки отдаются с `image/png`.
  - Push-уведомления (уведомления, когда сайт закрыт) в это не входят — раньше в этой же фазе договорились, что push делаем отдельно, если до этого дойдёт; сейчас только внутриприложенческий список.

Бэклог без конкретного плана: значок «район подтверждён» (не предлагать конкретный механизм без обсуждения — та же проблема лишнего барьера, что и с верификацией) и тематические комьюнити/каналы.
